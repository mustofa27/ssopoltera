# Integration Manual for External Applications

This document describes the current integration contract exposed by this SSO server.

It covers:
- Browser-based sign-in using OAuth 2.0 Authorization Code flow
- Access token exchange
- User profile retrieval
- Backchannel single logout
- Application registration requirements in the SSO admin panel

## 1. Integration Overview

External applications integrate with this server as OAuth clients.

Current flow:
1. Your application redirects the user to the SSO authorization endpoint.
2. The user authenticates in the SSO portal if not already signed in.
3. The SSO server verifies that the user is allowed to access your application.
4. The SSO server redirects back to your registered callback URL with either:
   - an authorization `code`, or
   - an OAuth error such as `access_denied`
5. Your backend exchanges the authorization code for an access token.
6. Your backend calls the `userinfo` endpoint to retrieve the authenticated user profile.

## 2. Registering an Application

Before integration, create the application in the SSO admin panel.

Required fields:
- Name
- Redirect URI

Optional but recommended fields:
- Description
- Logout URI
- Allowed scopes
- Allowed user types
- Role assignments
- Active status

Important behavior:
- `redirect_uri` must exactly match the value stored in SSO.
- `logout_uri` is used for backchannel logout notifications.
- `allowed_user_types` can restrict login to `student`, `employee`, or both.
- Role assignments determine which SSO users are allowed to access the application.
- The application must be active.

Credentials generated by SSO:
- `client_id`
- `client_secret`

Store both securely. Never expose `client_secret` in browser code.

## 3. Base Endpoints

Base domain for this environment: `https://sso.poltera.ac.id`.

- Authorization endpoint: `GET https://sso.poltera.ac.id/oauth/authorize`
- Token endpoint: `POST https://sso.poltera.ac.id/oauth/token`
- User info endpoint: `GET https://sso.poltera.ac.id/oauth/userinfo`
- User info endpoint: `POST https://sso.poltera.ac.id/oauth/userinfo`
- Backchannel logout trigger: `POST https://sso.poltera.ac.id/sso/backchannel/logout`

## 4. Authorization Flow

### 4.1 Redirect the user to SSO

Redirect the browser to:

```text
https://sso.poltera.ac.id/oauth/authorize
    ?response_type=code
    &client_id=YOUR_CLIENT_ID
    &redirect_uri=YOUR_REGISTERED_REDIRECT_URI
    &scope=openid%20profile%20email
    &state=RANDOM_CSRF_TOKEN
```

Required query parameters:
- `response_type=code`
- `client_id`
- `redirect_uri`

Optional query parameters:
- `scope`
- `state`

Current authorization rules enforced by SSO:
- Client application must exist and be active.
- `redirect_uri` must exactly match the registered redirect URI.
- User must already be authenticated in SSO or will be redirected to the SSO login page.
- User account must still pass active-session and password-expiry checks enforced by SSO.
- User must have access to the application through assigned roles.
- User type must be allowed by the application configuration.
- Requested scopes must be a subset of the application's allowed scopes if scopes are configured.

### 4.2 Successful authorization response

On success, SSO redirects the browser back to your callback URL:

```text
https://client.example.com/auth/sso/callback?code=AUTHORIZATION_CODE&state=RANDOM_CSRF_TOKEN
```

Returned parameters:
- `code`
- `state` if you provided `state`

### 4.3 Error authorization response

If authorization is denied, SSO redirects back to the same registered callback URL with:

- `error`
- `error_description`
- `state` if you provided `state`

Example:

```text
https://client.example.com/auth/sso/callback?error=access_denied&error_description=User+type+is+not+allowed+for+this+application.&state=RANDOM_CSRF_TOKEN
```

Possible error values from the current implementation:
- `invalid_client`
- `invalid_request`
- `invalid_scope`
- `access_denied`

## 5. Client Callback Handling

Your callback endpoint must handle both OAuth errors and successful authorization codes.

Recommended pattern:

```php
if ($request->query('error')) {
    abort(403, $request->query('error_description', 'OAuth authorization failed.'));
}

$expectedState = (string) $request->session()->get('sso.oauth_state', '');
$actualState = (string) $request->query('state', '');

if ($expectedState === '' || $actualState === '' || ! hash_equals($expectedState, $actualState)) {
    abort(403, 'Invalid OAuth state.');
}

$request->session()->forget('sso.oauth_state');

$authorizationCode = (string) $request->query('code', '');

if ($authorizationCode === '') {
    abort(400, 'Missing authorization code.');
}
```

Implementation notes:
- Generate and store the state value before redirecting the browser to SSO.
- Validate the returned `state` against your session value.
- Handle `error` before attempting token exchange.
- Only clear the stored state after successful validation.
- Perform token exchange from your backend, not from browser JavaScript.

## 6. Token Exchange

Exchange the authorization code from your backend.

Example request:

```bash
curl -X POST "https://sso.poltera.ac.id/oauth/token" \
  -H "Accept: application/json" \
  -d "grant_type=authorization_code" \
  -d "code=AUTHORIZATION_CODE" \
  -d "client_id=YOUR_CLIENT_ID" \
  -d "client_secret=YOUR_CLIENT_SECRET" \
  -d "redirect_uri=YOUR_REGISTERED_REDIRECT_URI"
```

Required form fields for authorization code grant:
- `grant_type=authorization_code`
- `code`
- `client_id`
- `client_secret`
- `redirect_uri`

Validation rules enforced by SSO:
- Client must exist and be active.
- `client_secret` must match exactly.
- `redirect_uri` must match exactly.
- Authorization code must belong to the same application.
- Authorization code must be unused.
- Authorization code must not be expired.
- Authorized user type must still be allowed at token-exchange time.

Successful response:

```json
{
  "access_token": "...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "...",
  "scope": "openid profile email"
}
```

Current token behavior:
- `access_token` is issued and stored in the SSO session table.
- `refresh_token` is issued and can be used to request a new access token.
- Refresh token rotation is enabled. Each successful refresh returns a new refresh token.

Current error responses:
- `401 {"error":"invalid_client"}`
- `400 {"error":"invalid_grant"}`
- `400 {"error":"invalid_grant","error_description":"Redirect URI mismatch."}`
- `400 {"error":"invalid_grant","error_description":"Refresh token expired."}`
- `403 {"error":"access_denied","error_description":"User type is not allowed for this application."}`

### 6.1 Refresh Access Token

Use this when your access token is expired or near expiry.

Example request:

```bash
curl -X POST "https://sso.poltera.ac.id/oauth/token" \
  -H "Accept: application/json" \
  -d "grant_type=refresh_token" \
  -d "refresh_token=YOUR_REFRESH_TOKEN" \
  -d "client_id=YOUR_CLIENT_ID" \
  -d "client_secret=YOUR_CLIENT_SECRET"
```

Required form fields for refresh-token grant:
- `grant_type=refresh_token`
- `refresh_token`
- `client_id`
- `client_secret`

Validation rules enforced by SSO:
- Client must exist and be active.
- `client_secret` must match exactly.
- `refresh_token` must belong to an active SSO session for the same application.
- `refresh_token` age must be within `SECURITY_SSO_REFRESH_TOKEN_TTL_MINUTES`.
- User type must still be allowed by the application configuration.

Successful refresh response:

```json
{
  "access_token": "...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "...",
  "scope": ""
}
```

Implementation notes:
- Replace both stored tokens in your client when refresh succeeds.
- If refresh returns `invalid_grant`, restart login with authorization code flow.

## 7. User Info Endpoint

Use the access token to retrieve the authenticated user profile.

Example request:

```bash
curl -X GET "https://sso.poltera.ac.id/oauth/userinfo" \
  -H "Authorization: Bearer ACCESS_TOKEN" \
  -H "Accept: application/json"
```

The endpoint accepts both `GET` and `POST`.

Successful response example:

```json
{
  "sub": "123",
  "name": "User Name",
  "email": "user@example.com",
  "avatar": null,
  "department": "Faculty of Engineering",
  "job_title": "Lecturer",
  "user_type": "employee",
  "employee_type": "lecturer",
  "nip": "19870001",
  "nrp": null,
  "roles": [
    {
      "id": 2,
      "name": "Lecturer",
      "slug": "lecturer"
    }
  ],
  "organization": {
    "department": "Faculty of Engineering",
    "program_study": "Informatics",
    "support_unit": null
  }
}
```

Current profile fields returned:
- `sub`
- `name`
- `email`
- `avatar`
- `department`
- `job_title`
- `user_type`
- `employee_type`
- `nip`
- `nrp`
- `roles[].id`
- `roles[].name`
- `roles[].slug`
- `organization.department`
- `organization.program_study`
- `organization.support_unit`

Field source notes:
- `roles` contains only the roles that grant the user access to the current client application tied to the presented access token.
- For users linked to Microsoft 365, `name`, `email`, `department`, and `job_title` are treated as Microsoft-managed profile data in SSO.
- These fields can be refreshed by SSO from Microsoft identity data and should be treated as mutable by client applications.
- Use `sub` as your stable user identifier in your local database.

Error response:
- `401 {"error":"invalid_token"}` when token is missing, expired, or unknown

## 8. Access Rules That Affect Login

Even when your OAuth request is valid, SSO may deny authorization if the user is not permitted to access your application.

Current checks:
- The application is active.
- The user has at least one assigned role that is also assigned to the application.
- The user's `user_type` matches the application's allowed user types when restrictions are configured.

Practical example:
- If your application is restricted to `employee`, a `student` user will receive `access_denied`.
- If your application has no allowed user type restrictions configured, user type does not block access.

## 9. Backchannel Logout Trigger

External applications can ask SSO to revoke the user's SSO sessions and notify other connected applications.

Endpoint:

```text
POST /sso/backchannel/logout
```

JSON request body:

```json
{
  "client_id": "YOUR_CLIENT_ID",
  "user_id": 123,
  "timestamp": 1710000000
}
```

You may send `user_email` instead of `user_id`.

At least one of these must be present:
- `user_id`
- `user_email`

### 9.1 Signature requirement

You must send header `X-SSO-Signature`.

Signature algorithm:
- HMAC SHA-256
- Message: raw JSON request body exactly as sent
- Secret key: your `client_secret`

Pseudo-code:

```text
signature = HMAC_SHA256(rawJsonBody, client_secret)
```

Example header:

```text
X-SSO-Signature: <hex_hmac_signature>
```

### 9.2 Timestamp requirement

The request must include `timestamp` within the allowed skew window.

Current server setting:
- `SECURITY_SSO_BACKCHANNEL_MAX_SKEW_SECONDS`
- Default: `300`

### 9.3 Successful response

```json
{
  "message": "Single logout completed.",
  "data": {
    "revoked_sessions": 3,
    "notified_applications": 2,
    "failed_notifications": 0
  }
}
```

### 9.4 Possible error responses

- `401 {"message":"Invalid client."}`
- `401 {"message":"Invalid signature."}`
- `404 {"message":"User not found."}`
- `422 {"message":"Either user_id or user_email is required."}`
- `422 {"message":"Request timestamp is out of allowed window."}`

## 10. Receiving Logout Notifications in Your Application

If your application has `logout_uri` configured, SSO can notify your application when a single logout event occurs.

The SSO server sends:
- `POST` request to your configured `logout_uri`
- `Content-Type: application/json`
- Header `X-SSO-Event: single_logout`
- Header `X-SSO-Signature: <hex_hmac_signature>`

Payload shape:

```json
{
  "event": "sso.single_logout",
  "user_id": 123,
  "user_email": "user@example.com",
  "application_slug": "your-app",
  "occurred_at": "2026-04-01T08:00:00+00:00",
  "source": "https://sso.poltera.ac.id",
  "method": "backchannel"
}
```

Signature verification rule:
- Compute HMAC SHA-256 of the raw JSON body using your `client_secret`
- Compare it to `X-SSO-Signature`

Client application recommendations:
- Verify the signature before taking action.
- Revoke the local session immediately after verification.
- Return HTTP 200 quickly.
- Avoid expensive processing inside the request cycle.

## 11. Dashboard Launch Behavior

In the SSO dashboard, the `Open App` button uses the base URL derived from the registered `redirect_uri`.

Example:
- Registered redirect URI: `https://client.example.com/auth/sso/callback`
- Dashboard launch URL: `https://client.example.com`

This means:
- Your application should be reachable from the same origin as the registered callback URL.
- The dashboard does not redirect users directly to the callback path.

## 12. Security Recommendations

- Use HTTPS in staging and production.
- Keep `client_secret` on the server side only.
- Always validate OAuth `state` in the client callback.
- Handle `error` and `error_description` in the callback before token exchange.
- Log failed token exchanges, callback errors, and logout signature failures.
- Rotate `client_secret` after regeneration in the SSO admin panel.
- Keep your registered `redirect_uri` exact and stable.

## 13. Integration Checklist

- Register the application in SSO.
- Save the generated `client_id` and `client_secret`.
- Configure the exact callback URL as `redirect_uri`.
- Configure `logout_uri` if you want backchannel logout notifications.
- Assign the correct SSO roles to the application.
- Configure allowed user types if needed.
- Implement state generation and validation.
- Implement callback handling for both `code` and `error` responses.
- Exchange authorization code from the backend.
- Use the access token to call `userinfo`.
- Persist `sub` as the canonical external user key in your application.
- Implement backchannel logout signing and verification if your app participates in single logout.

## 14. Current Limitations

- Authorization Code flow is implemented.
- No OpenID discovery document is exposed.
- No JWKS endpoint is exposed.
- Access tokens are opaque session-backed tokens, not JWTs.
- No OAuth endpoint currently returns organization head assignment fields (head of department/support unit/program study).
